### Document Type Badge Colors (for chain badges in unified list)

| Type | Prefix | Background Class | Text Class |
|------|--------|-----------------|-----------|
| QUOTATION | QT | `bg-slate-100 dark:bg-slate-800` | `text-slate-700 dark:text-slate-300` |
| INVOICE | INV | `bg-blue-100 dark:bg-blue-800` | `text-blue-700 dark:text-blue-300` |
| RECEIPT | RCT | `bg-green-100 dark:bg-green-800` | `text-green-700 dark:text-green-300` |
| DELIVERY_NOTE | DLV | `bg-amber-100 dark:bg-amber-800` | `text-amber-700 dark:text-amber-300` |

**Rationale:** Distinct colors per document type enable instant visual differentiation in chain badges. The `overdue` invoice status uses `red` (not `destructive`) to distinguish from the void/rejected destructive palette while still conveying urgency. `delivered` for delivery notes reuses `green` since it is a positive terminal state equivalent to `confirmed` and `paid`.

## Interaction Contracts

### Conversion Buttons (NEW -- Phase 6 core interaction)

| Source Page | Button Label | Visible When | Behavior | Button Variant |
|------------|-------------|-------------|----------|----------------|
| Quotation detail | สร้างใบแจ้งหนี้ | `effectiveStatus === "accepted"` | One-click: submit form with sourceDocumentId -> server action creates invoice atomically -> redirect to `/apps/invoice/{newId}` | `default` |
| Quotation detail | สร้างใบส่งของ | `effectiveStatus === "accepted"` or `effectiveStatus === "converted"` | One-click: create delivery note from quotation data -> redirect to `/apps/delivery-note/{newId}` | `outline` |
| Invoice detail | สร้างใบเสร็จ | `effectiveStatus !== "voided"` and `effectiveStatus !== "draft"` | One-click: create receipt with default payment info -> redirect to `/apps/receipt/{newId}` | `default` |
| Invoice detail | สร้างใบส่งของ | `effectiveStatus === "sent"` or `effectiveStatus === "overdue"` | One-click: create delivery note from invoice data -> redirect to `/apps/delivery-note/{newId}` | `outline` |

**Implementation pattern:** Each conversion button is a `<form>` with a hidden `sourceDocumentId` input and a `formAction` pointing to the conversion server action. The button is a submit button. No dialog, no pre-filled form. Click submits -> action creates -> redirect.

```
[Conversion buttons area -- within detail-actions.tsx, after status buttons]
  |-- Separator (visual: border-l h-6 mx-1, only on sm: breakpoint)
  |-- Button: "สร้างใบแจ้งหนี้" with FileText icon (h-4 w-4 mr-2)
  |-- Button: "สร้างใบส่งของ" with Truck icon (h-4 w-4 mr-2)
```

**Loading state:** Button shows `Loader2` spinner + Thai loading text. Button `disabled` during submission. Toast on error only (success redirects away from page).

## Responsive Behavior

Inherited from Phase 5. Same `sm:` breakpoint strategy.

| Breakpoint | Behavior |
|-----------|----------|
| Mobile (<640px, below `sm:`) | Line items stack vertically. Cards full width. Tables scroll horizontally. Action buttons and conversion buttons stack vertically. Filter row wraps. Chain badges truncate to type prefix only (QT, INV, RCT, DLV). |
| Tablet/Desktop (>=640px, `sm:`) | Line items in horizontal grid. Cards at natural width. Tables fit without scroll. Action buttons inline. Full chain badge display. Filters in single row. |

**Chain badge mobile truncation:** On mobile, chain badges show only the type prefix (e.g., "QT", "INV") without the number suffix to save horizontal space. Full number shown in tooltip on tap.
